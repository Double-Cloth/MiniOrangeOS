# 开发者审查与心得

本文档在每个里程碑结束时记录实际阅读、理解、问题修正和尚需学习的内容。任务级事实记录在 docs/task-reports，来源记录在 docs/provenance.md。

## M0 完成

已确认的工程原则：

- Windows 目录是唯一工作树，Windows Git 是唯一 Git。
- WSL 只提供 Linux 构建和测试语义。
- 按阶段建分支、先测试、同步文档、验收后 no-ff 合并。
- 未运行的测试不能写成 PASS。

M0 已完成并验证以下能力：

- WSL 发行版创建、验证、备份和定向删除边界；
- i686-elf 工具链的组成和隔离路径；
- GNU Make 依赖、并行和增量构建行为；
- QEMU 串口、debug-exit、超时和 GDB 回环调试链。

## T01 心得

- 真实 Ubuntu 的 `/etc/os-release` 是受信相对 symlink，安全校验应绑定精确目标，而不是一律拒绝 symlink。
- T00 由 root 创建的只读中间目录可以安全存在，但最终环境根必须属于目标用户；路径逐级 ownership 比笼统 `chown` 更可审计。
- 可恢复容器生命周期需要同时绑定锁、intent、state、镜像 ID 与标签，单靠镜像名称不足以证明 ownership。
- rootless Podman 的 runroot 长度和 overlay subuid 清理只有真实集成会暴露；fake backend 必须模拟这些失败边界。
- WSL2 验收证明用户态与 Microsoft 内核组合，不等于原生 Linux CI 证据。
- 可恢复清理不能假设 `--rm`、image 或 builder 始终与 state 同步；必须在每个 mutation 阶段重新枚举并复核 ownership，把“可信且已缺失”与“同名 foreign replacement”明确区分。
- 固定归档哈希只证明下载来源，不能证明复用的解压树未漂移；可复现缓存还需绑定目录 mode、symlink、文件内容和 hardlink 拓扑，并在执行源码前复核。
- 安全的实例身份不能来自调用者可覆盖的环境变量；Windows Lxss 注册事实需要落为 Linux 内 root-owned、不可由普通用户改写的最小身份记录。
- 仅在 pathname 上重复 `stat` 不能封闭 target-owned 目录的 TOCTOU；敏感写入应锚定 nofollow 目录 FD，FD 必须 CLOEXEC，并为不可捕获终止设计严格、可审计的下一次恢复协议。

## T02 心得

- `clean` 也是高风险产品功能：可配置输出目录必须有不可复制的归属证明，并在校验、隔离和递归删除全过程绑定同一 inode。
- 原子 `replace` 只保护提交瞬间；输入组件、输出父目录和临时文件还需要 dirfd/nofollow、前后身份复核与失败清理共同封闭竞态。
- 固定镜像上限不能替代流式实现；稀疏文件测试能同时验证内存上限、零区语义和确定性。

## T03 心得

- 串口 PASS 只是协议输入，成功还必须同时满足完整有序状态机、无 FAIL、QEMU 真实退出和精确 debug-exit 状态。
- 进程清理要绑定本次 leader/PGID，并在回收前保留 leader 身份；subreaper 才能覆盖容器中双重 fork 的孤儿后代。
- 路径安全需要把镜像、构建根和日志目录绑定到已验证 FD；原子 rename 后仍要核对最终 inode，且 DrvFS 的可见性语义必须由真实工作树测试验证。

## T10 心得

- DAP 的 segment:offset 不回绕不等于物理 DMA 边界安全；跨 `0x10000` 的连续区域需要拆成两个 BIOS 请求。
- Stage 1 自己打印 `loader loaded` 不能证明交接成功；最小 Stage 2 探针应真实核验寄存器、标志和退出握手。
- 由单一布局生成汇编常量仍属于构建产品功能，必须继承 marker、nofollow、严格 schema、原子提交和源码零副作用边界。

## T11 心得

- ELF32 只是链接容器，不会改变 CPU 当前位宽；实模式代码必须明确使用 16 位指令，并由链接断言保证入口和绝对地址不越过 `0xFFFF`。
- BIOS wrapper 的可靠性不仅是发出 `int`：调用约定、寄存器和 flags 必须明确，并通过直接链接产品实现的真实 QEMU fixture 验证，静态反汇编匹配不足以排除假绿。

## P1 心得

- 高半 ELF 可以先按 `p_paddr` 装入并从物理入口执行，但分页前的早期路径只能依赖相对寻址和显式物理指针；绝对高半地址必须等页表生效后使用。
- E820 原始条目可能重叠，Loader 装载校验必须让非 type 1 保留区域优先，不能因为找到一个包含目标的 type 1 条目就立即接受。
- 镜像布局的“单一来源”不仅约束 Stage 1；Kernel LBA 和区域上限也必须由同一生成文件交给 Stage 2，并进入构建依赖图。
- 通用 QEMU runner 的故障注入合同与产品启动链验收职责不同；保留 `test-qemu` 验证 runner，再用 `test-boot-qemu` 验证真实产品成功与损坏镜像路径，能避免两类证据互相伪装。

## P2 心得

- 页表和切换后的内核栈不能放入普通 `.bss`：高半入口清零 `.bss` 时会破坏当前地址空间或栈。独立 NOBITS 启动区既保留零初始化语义，又明确排除清零范围。
- 早期阶段让 PDE 0 与 PDE 768 共享同一页表，可以在设置 CR0.PG 后保持当前物理指令流有效，再通过一次显式跳转进入高半地址；低端恒等映射要等正式 VMM 接管后再回收。
- 串口是自动化权威输出，VGA 是交互输出；让格式化层扇出到两个字符设备，可以避免日志内容和显示内容各自实现一套格式化规则。
- COM1 的轮询即使在 panic 路径也必须有上限，否则“输出故障信息”本身可能成为不可诊断的永久阻塞点。
- Loader 临时 GDT 不能作为内核长期隐式依赖；内核重建表后必须同时重载数据段和 `SS`，再用 far jump 刷新 `CS`，并在真实分页/高半执行流中验证。
- 异常入口必须先把“CPU 自动压入错误码”和“无错误码”两类向量归一化，再交给 C；否则同一个 trap frame 在不同向量下字段会整体错位，日志中的 EIP 也不可信。
- 故障注入开关属于构建输入，也必须像路径变量一样失败关闭；只允许精确 `0/1`，并用独立构建目录避免编译参数变化被增量缓存掩盖。
- PIC 初始化应先屏蔽所有 IRQ，再在 IDT 和具体驱动就绪后逐项放开；“先开中断再补处理器”会把偶发硬件事件变成不可复现的启动故障。
- PIT 可观察性应使用一次性 tick 里程碑，而不是每 tick 写串口；后者会把中断处理时间和串口带宽混入时钟行为，掩盖真实调度问题。
- PS/2 控制器和键盘设备是两层状态机：控制器自检通过不代表设备已开始扫描，必须等待 `F4` 的 `FA` ACK 后再放开 IRQ1。
- 键盘中断不能把“输入过快”升级成内核崩溃；固定环形缓冲满时丢弃新字符，并把消费接口保持为非阻塞，后续再由等待队列提供阻塞语义。

## P3 心得

- E820 的 type 1 与保留条目可能重叠，PMM 必须先释放 usable、再覆盖 reserved；按输入顺序边遍历边决定最终状态会让结果依赖 BIOS 条目顺序。
- 可用范围只能释放完全包含的页，保留范围必须保留任何相交页，因此两类范围的页对齐方向相反，不能复用同一个“对齐范围”辅助函数。
- 单独的 `allocatable` bitmap 让 `free` 能区分“已分配页”和“永远不可释放的保留页”，避免只靠 used bit 把内核/BIOS 页误当成普通已分配页。
- 递归页表让任意 PDE/PTE 都有稳定虚拟地址，但新页表物理页在挂入页目录前仍不可直接访问；保留一个受控 scratch PTE 可先清零页面，再发布 PDE。
- 移除低端恒等映射后，Loader 传入的 Boot Info 低地址指针立即失效，因此正式 VMM 初始化必须先消费完所需字段，并禁止后续保存或解引用该指针。
- 仅把 PTE 标为只读不足以保护 Ring 0 写入；必须同时设置 CR0.WP，并通过链接符号按页收紧 text/rodata，data、bss、启动页表和栈保持可写。
- Heap 扩展跨越 PMM 分配与 VMM 映射两个资源域；任一页失败时必须逆序解除已映射页并归还物理页，不能把半成功扩展发布到链表。
- first-fit 的块头本身会制造碎片；只有剩余空间至少容纳块头和一个对齐后的最小 payload 时才拆分，释放时同时向前、向后合并才能恢复连续大块。
- magic 只能检测“已经访问到的块头”；链表遍历还必须先校验相邻块地址处于已映射堆区，再解引用反向指针，否则损坏元数据会先触发不可控 page fault。
- 构造非当前用户页目录时需要临时映射其物理页；复制高半 PDE 必须排除临时工作窗口，否则新页目录会保留指向已回收工作页表的悬空引用。
- usercopy 不能只检查首地址和末地址：每一页的 PDE/PTE 有效 U/S、R/W 权限都必须满足；字符串则逐页验证到 NUL，不能因 `max_len` 之后的未访问页未映射而误拒绝。
- page fault 的 U/S 错误码位提供故障来源边界；P3 只为用户故障提供可注册接管点，P4 才能在具备当前进程和调度器后实现“只终止当前进程”。

## P4 心得

- TSS descriptor 与普通代码/数据描述符不能共享 4 KiB granularity/32-bit segment flags；TSS 使用 byte granularity、limit=`sizeof(TSS)-1`，并把 I/O bitmap offset 放到 limit 之外以拒绝用户 I/O。
- `ltr` 只负责装载 TSS selector；每次切换到不同进程前仍必须更新 `esp0`，否则 Ring 3 中断会落到旧进程的内核栈。
- 首次线程没有真实的 C caller，必须人工构造与汇编 pop/ret 顺序完全一致的初始栈，并保留一个伪返回地址满足 C 栈入口形状；trampoline 自身不得正常返回。
- 关中断后切到新线程时，旧线程的 `irq_restore` 尚未执行；因此首次 trampoline 必须按创建时保存的 IF 状态恢复中断，不能无条件 `sti`。
- IRQ 内抢占会把尚未返回的完整中断调用链留在旧线程内核栈上；切回后必须从同一个 `context_switch` 返回并最终 `iretd`，不能只保存一个普通函数级寄存器快照就丢弃 IRQ 栈。
- 调度必须在 PIC EOI 之后发生；否则新线程恢复 IF 后，IRQ0 仍处于 in-service，会让后续 tick 与嵌套中断行为失真。
- 用户页目录不能从“当前页目录”复制高半 PDE：当前页目录可能是创建较早的用户快照，缺少后来扩展的 Heap 等内核映射。应保存主内核页目录，并在每次 CR3 激活前从它刷新共享 PDE；临时映射窗口必须在关中断区间使用且不能复制进目标地址空间。
- Ring 3 的 `int 0x80` 不只多压入 EIP/CS/EFLAGS，还因 CPL 变化压入用户 ESP/SS；syscall stub 必须在 `pushad` 之后另存段寄存器，才能保持 C trap frame 的通用寄存器/vector/error/EIP 布局，并在 `iretd` 前恢复用户段。
